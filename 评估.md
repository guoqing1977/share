这是一个可直接落地执行的跨平台性能评估流程，涵盖环境采集、微基准、应用全栈压测、高负载干扰、NUMA 延迟对比等环节。只需在两套环境（x86+RHEL 与 海光C86+麒麟V10）上同步执行相同命令即可获得对比数据。

---

评估前核心原则

· 硬件一致：两套环境 CPU 核心数、内存容量/频率、磁盘类型、网络带宽对齐
· 软件镜像：JDK 版本、工具版本、系统内核参数完全一致
· 脚本复用：所有脚本和 JMeter 测试计划写一次，两平台共用

阶段 3：微基准测试（实施时长 20 分钟）

3.1 CPU & 内存微基准（sysbench）

```bash
# CPU 素数计算（单线程）
sysbench cpu --cpu-max-prime=20000 --threads=1 run | tee cpu_single.log

# CPU 多线程（等于物理核数）
sysbench cpu --cpu-max-prime=20000 --threads=$(nproc) run | tee cpu_multi.log

# 内存读写带宽（块大小与总量建议设大）
sysbench memory --memory-block-size=1M --memory-total-size=100G run | tee memory_bw.log
```

采集指标：单/多核 events per second、内存 MiB/sec。

3.2 系统调用开销微基准（可选）

保存以下 C 程序为 clock_test.c，在两平台编译运行：

```c
#include <stdio.h>
#include <time.h>
#define LOOPS 1000000
int main() {
    struct timespec ts = {0, 1};
    for (int i = 0; i < LOOPS; i++)
        clock_gettime(CLOCK_MONOTONIC, &ts);
    return 0;
}
```

```bash
gcc -O2 clock_test.c -o clock_test && time ./clock_test
```

对比 real 时间。海光平台下某些系统调用耗时可能显著偏高。

阶段 4：Java 微基准测试（JMH 示例）

4.1 准备 JMH 项目（Maven）

```bash
mvn archetype:generate \
  -DgroupId=com.eval \
  -DartifactId=jmh-test \
  -DarchetypeGroupId=org.openjdk.jmh \
  -DarchetypeArtifactId=jmh-java-benchmark-archetype \
  -DinteractiveMode=false
cd jmh-test
```

修改生成的 MyBenchmark.java 为模拟业务计算的斐波那契：

```java
package com.eval;
import org.openjdk.jmh.annotations.*;
import java.util.concurrent.TimeUnit;

@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
@State(Scope.Thread)
public class MyBenchmark {
    @Benchmark
    public long fib35() {
        return fib(35);
    }
    private long fib(int n) {
        if (n <= 1) return n;
        return fib(n-1) + fib(n-2);
    }
}
```

4.2 执行并记录

```bash
mvn clean package
java -jar target/benchmarks.jar -f 1 -wi 5 -i 5 -rf csv -rff jmh_fib35.csv
```

采集指标：Average Time (Score), 错误范围。

阶段 5：业务模拟应用 & 基线压测（实施时长 30 分钟）

5.1 蒙特卡洛 Java 服务

在两平台相同目录创建 MonteCarloServer.java，内容如下：

```java
import com.sun.net.httpserver.*;
import java.io.*;
import java.net.InetSocketAddress;
import java.util.Random;
import java.util.concurrent.Executors;

public class MonteCarloServer {
    public static void main(String[] args) throws Exception {
        HttpServer server = HttpServer.create(new InetSocketAddress(8080), 0);
        server.setExecutor(Executors.newFixedThreadPool(50));
        server.createContext("/work", new WorkHandler());
        server.start();
        System.out.println("Server started on 8080");
    }

    static class WorkHandler implements HttpHandler {
        private final Random rand = new Random();
        @Override
        public void handle(HttpExchange exchange) throws IOException {
            long start = System.nanoTime();
            double p = rand.nextDouble();
            int n = (p < 0.70) ? 25 : (p < 0.95) ? 35 : 42;
            fib(n);
            long ms = (System.nanoTime() - start) / 1_000_000;
            String resp = "{\"type\":\"" + (n<=25?"light":n<=35?"medium":"heavy") + "\",\"ms\":" + ms + "}";
            exchange.getResponseHeaders().set("Content-Type", "application/json");
            exchange.sendResponseHeaders(200, resp.length());
            OutputStream os = exchange.getResponseBody();
            os.write(resp.getBytes());
            os.close();
        }
        static long fib(int n) {
            if (n <= 1) return n;
            return fib(n-1) + fib(n-2);
        }
    }
}
```

启动方式（绑定 NUMA 节点 1 确保基线不受干扰）：

```bash
numactl --cpunodebind=1 --membind=1 java MonteCarloServer.java &
```

阶段 6：高负载干扰测试（实施时长 30 分钟）

6.1 场景设计

场景 stress-ng 绑定 Java 绑定 模拟目标
基线（已测试） 无 node1 本地 洁净环境
CPU 竞争 node0 100% CPU node0 本地内存 同节点争抢核与缓存
远端内存 node0 100% CPU node0 CPU + node1 内存 最差情况：跨节点访存

注意：若环境只有单 NUMA 节点，可跳过远端内存场景，但可通过 --membind 模拟。此处要求硬件有至少 2 个 NUMA 节点。

6.2 自动化干扰测试脚本

创建 interference_test.sh：

```bash
#!/bin/bash
JMETER_BIN=$1
STRESS_DUR=300
COOLDOWN=30

echo "=== 场景A: 同节点 CPU 竞争 ==="
numactl --cpunodebind=0 --membind=0 stress-ng --cpu 0 --cpu-load 100 --timeout ${STRESS_DUR}s &
STRESS_PID=$!
sleep 5
numactl --cpunodebind=0 --membind=0 java MonteCarloServer.java &
APP_PID=$!
sleep 20
$JMETER_BIN -n -t baseline.jmx -l /tmp/result_cpu_contend.jtl
kill $APP_PID $STRESS_PID 2>/dev/null
sleep $COOLDOWN

echo "=== 场景B: 远端内存访问 ==="
numactl --cpunodebind=0 --membind=0 stress-ng --cpu 0 --cpu-load 100 --timeout ${STRESS_DUR}s &
STRESS_PID=$!
sleep 5
numactl --cpunodebind=0 --membind=1 java MonteCarloServer.java &   # CPU在0，内存在1
APP_PID=$!
sleep 20
$JMETER_BIN -n -t baseline.jmx -l /tmp/result_remote_mem.jtl
kill $APP_PID $STRESS_PID 2>/dev/null

echo "干扰测试完成"
```

执行：

```bash
chmod +x interference_test.sh
./interference_test.sh $JMETER_HOME/bin/jmeter
```

阶段 7：结果汇总与分析

7.1 提取 P50/P90/P99 延迟（从 JMeter JTL）

JMeter 原生不支持直接计算百分位，可使用 Python 或 awk。这里提供 awk 一行的近似值（需排序）：

```bash
# 提取响应时间列（假设第2列为elapsed）
for file in /tmp/result_baseline.jtl /tmp/result_cpu_contend.jtl /tmp/result_remote_mem.jtl; do
  echo "===== $file ====="
  awk -F',' 'NR>1 {print $2}' $file | sort -n | awk '
  { all[NR]=$0 }
  END {
    print "P50: " all[int(NR*0.5)];
    print "P90: " all[int(NR*0.9)];
    print "P99: " all[int(NR*0.99)];
    print "Max: " all[NR];
  }'
done
```

7.2 建立对比表

平台A：x86 + RHEL

同样填写。重点关注：

· 微基准算力差距百分比
· P99 延迟恶化倍率（远端内存场景对比基线）
· 在 100% 压力下 Java 服务 CPU 使用率是否异常偏高（用 top 观察）

7.3 高阶定位（发现问题时）

若海光平台 P99 异常高：

```bash
# 采样热点函数（压测时执行）
perf top -p $(pgrep -f MonteCarloServer)

# 查看 NUMA 内存分布
numastat -p $(pgrep -f MonteCarloServer)
```

观察是否有大量 other_node 分配，或内核态调用（如 native_queued_spin_lock_slowpath）占比过高，这通常指示底层锁竞争或平台差异。

---

最终决策建议

1. 延迟敏感业务：重点关注“远端内存 P99 恶化倍率”，若海光平台该倍率 >2x，应慎重。
2. 吞吐优先业务：对比基线 TPS 与 CPU 核心数，计算“每核吞吐”，评估单位算力成本。
3. 运维兼容性：海光 C86 完全兼容 x86 指令集，应用可直接迁移，降低改造成本。可优先迁移非核心业务，积累运行经验。

将所有表格数据填入后，即可清晰看到两套平台在纯净性能、抗干扰能力、NUMA 影响下长尾延迟三个层次的真实差异，为技术选型提供量化依据。
